How this works

A simple operating system behind the website

The key idea is that staff should update wine information in a simple admin table, not in website code. The website then reads the approved output and shows customers the right version of the range.

What staff should edit

The product tables

Catalogue, pricing, stock, images, and publishing settings. In a real setup, this would be Airtable or a spreadsheet-style admin, not HTML files.

What staff should not edit

The page layout code

HTML and CSS control the look of the site. They should change only when you want a redesign, not when you want to update a wine, a price, or stock.

Daily workflow

What a manager would actually do

1

Add or update the wine

Fill in the wine identity, region, producer, grapes, and supplier details.

2

Set price and stock

Enter bottle price, take-home price, glass price if needed, and current stock.

3

Choose website status

Decide whether the wine is internal only, info only, collection only, delivery only, or both.

4

Check image and notes

Add the approved bottle image and the public tasting note or Daniel recommendation.

5

Publish

The website and Shopify feed update from the approved records rather than from manual copy-paste.

Important rule

Not every wine in stock should be sold online

Crush has a broader in-bar range than its take-home range. This model keeps those separate so the website only shows what you want customers to see and buy.

Editing answer

Should this be edited in HTML?

For normal use: no. HTML is for designers or developers. Day-to-day editing should happen in Airtable or a spreadsheet-like admin. This demo uses data files because it is a prototype, but the intended live workflow is no-code for regular updates.
